# RainRoute — 騎車雨區路線 App 設計規格書

- **日期**：2026-07-21
- **狀態**：設計定案（待實作）
- **實作對象**：本規格為自足文件，可直接交給實作者（含 Fable 5）依此開發

---

## 1. 背景與動機

使用者騎機車通勤／出遊，出發前想知道「這趟路線上現在（或沿途時間）有沒有下雨」，以決定要不要穿雨衣。現有 Google 地圖能規劃路線，但**無法把路線疊到即時降雨雷達上**判斷是否會淋雨。

本 App 的核心價值 = **路線規劃 + 即時雨區疊圖 + 一句「要不要穿雨衣」的結論**。

地理範圍限定**台灣境內**，因此採用中央氣象署（CWA）的免費、高精度雷達資料。

---

## 2. 目標與非目標

### 目標
- 設定出發地與目的地，規劃出一條路線
- 在地圖上把路線與**即時降雨雷達回波**疊在一起顯示
- 最上方給一句明確結論：**「建議穿雨衣」／「不需要穿雨衣」**
- 全部使用開源／免費資料源，不綁信用卡

### 非目標（本版不做）
- 語音逐步導航（turn-by-turn navigation）
- 帳號系統、社群功能
- 上架 App Store／Google Play（先自用）
- iOS 以外的最佳化（先以使用者自己的手機為主）

## 3. 關鍵決策紀錄（brainstorming 結論）

| 議題 | 決定 | 理由 |
|------|------|------|
| 下雨判斷 | **即時 + 沿途時間預測，分階段** | 先做即時（MVP），再加時間預測 |
| 平台 | **原生手機 App（Expo / React Native）** | 可直接跑自己手機、不必上架 |
| 地理範圍 | **台灣境內** | 採用 CWA 免費高精度雷達 |
| 地圖／路線 | **全開源方案** | 不綁信用卡、用戶成長不會帳單爆炸、核心價值在雨區疊圖而非頂級路線 |
| 結果呈現 | **地圖疊雨區 + 一句結論** | 直觀 |
| 後端部署 | **先掛 NUC（B），好用再搬雲端（A）** | 先用既有設備最快看到成果；架構須設計成可無痛搬遷 |
| 機車路線 | **以汽車路網代替** | 開源路線引擎無專門機車模式，台灣機車走的路與汽車幾乎重疊 |

## 4. 整體架構

```
┌─────────────────────┐        ┌──────────────────────────────┐
│   手機 App (Expo)     │ <────> │   後端 API (Python FastAPI)    │
│   - MapLibre 地圖     │  HTTP  │   - /geocode  地址 → 座標       │
│   - 畫路線 折線        │        │   - /route    起訖 → 路線折線    │
│   - 疊雨區 raster      │        │   - /rain     路線 → 雨區判斷    │
│   - 結論 banner       │        │   - 快取 CWA 雷達圖 (~10 分)     │
└─────────────────────┘        └───────────────┬──────────────┘
                                               │ 所有金鑰只留在後端
                              ┌────────────────┼─────────────────┐
                         CWA 雷達回波      OpenRouteService      Nominatim
                         (O-A0058)         (路線規劃)            (地址→座標)
```

### 為什麼要有後端
1. **金鑰安全**：CWA、OpenRouteService 的 key 全部留在伺服器，不外洩；未來要賣，用戶不需自己申請 key。
2. **對齊實作者強項**：後端用 **Python FastAPI**。
3. **雨區運算集中**：雷達圖下載、快取、像素取樣、疊圖產生都在 Python 端做，比在手機端做省事很多。
4. **未來擴充點**：之後要加「AI 給穿衣／出行建議」的 agent 層（如 LangGraph），掛在後端最順。

### 部署遷移原則（B → A 無痛搬遷）
- 後端所有外部依賴（CWA key、ORS key、對外網址、快取路徑、監聽 port）**一律用環境變數**，不可寫死。
- App 端的後端 base URL 也用設定檔／環境變數，換位置只改一個值。
- 後端不依賴 NUC 本機特有狀態（不用本機 GPU、不用本機檔案系統做持久化，快取用可替換的抽象）。
- **Phase 1 部署 = NUC**：FastAPI 直接跑在使用者既有的 NUC 上（本來就開著）。
- **未來部署 = 免費雲端**（Render / Railway / Fly.io / Cloud Run，閒置自動休眠）。因為上面原則，搬過去只需改環境變數 + 部署設定。

### 外網連線（重要）
本 App 的使用情境是**人在外面**（出門前／騎車途中）查雨，此時手機在行動網路上、不在家裡區網。因此後端掛 NUC 時，**必須讓手機從外網連得到 NUC**，否則出了家門即失效。

- **採用 Tailscale**（預設方案）：NUC 與手機各安裝一次，組成私有 mesh 網路；手機在任何地方都能用 NUC 的 Tailscale IP（`100.x.x.x`）連到後端。免費、免開 port、加密。
- App 端 `BACKEND_BASE_URL` 指向 NUC 的 Tailscale IP。未來搬雲端時只改此值，符合遷移原則。
- 備選：Cloudflare Tunnel（公開網址）、路由器 port forwarding + DDNS（較不安全，不推薦）。

## 5. 核心機制：路線雨區判斷（本 App 靈魂）

採用**雷達回波圖像素取樣法**。

### 資料源：CWA「雷達整合回波圖-臺灣」
- 開放資料平台：https://opendata.cwa.gov.tw/ （免費註冊會員 → 取得授權碼 API key）
- 資料集代碼族：`O-A0058`（雷達整合回波圖-臺灣）
  - 實作時請確認並優先選用**「無地形」版本**，因為純回波、無地形底圖，像素取樣時顏色不會被地形圖干擾。（`O-A0058` 系列含 `-001/-003/-006` 等子項，實作第一步先用 Swagger 文件確認各子項的內容與回傳格式，挑選「無地形、含地理範圍資訊」者。）
- 檔案下載 URL 格式：
  `https://opendata.cwa.gov.tw/fileapi/v1/opendataapi/{dataid}?Authorization={apikey}&format={format}`
- 已知地理範圍（此為判斷像素座標的關鍵）：
  - 經度 **115.00 – 126.50**
  - 緯度 **17.75 – 29.25**
  - 影像解析度約 **3600 × 3600** 像素
  - （實作時仍應以該資料回傳的 GeoInfo／metadata 為準，不要寫死；上述數值作為預期值與驗證基準。）

### 判斷流程
1. **取得路線折線**：呼叫 `/route`，得到一串座標點 (lat/lng) 組成的 polyline。
2. **沿線取樣**：沿折線每隔固定距離（建議 **500 公尺**，可調）取一個取樣點。
3. **座標 → 像素**：用雷達圖的地理範圍，把每個取樣點的經緯度線性換算成圖上像素 (x, y)。
   - `x = (lng - LeftLon) / (RightLon - LeftLon) * width`
   - `y = (TopLat - lat) / (TopLat - BottomLat) * height`
4. **讀顏色 → 強度**：讀該像素顏色，對照 CWA 回波強度色階（dBZ）換算降雨強度，分級：
   - 無雨 / 小雨 / 中雨 / 大雨
   - 對照表：實作時依 CWA 官方色階建立「RGB → dBZ 區間 → 降雨分級」查表；透明／無回波像素視為無雨。
5. **產生結論**：
   - 規則（MVP 版，可調）：**路線上任一取樣點達「中雨」以上 → 建議穿雨衣**；否則不需要。
   - 回傳時附帶：有雨的路段位置（哪幾個取樣點）、最大強度分級，供 App 顯示。

### 地圖疊圖
- 同一張雷達回波圖，**半透明**疊在地圖上台灣範圍（依其地理 bbox 對齊），作為 MapLibre 的 raster image source。
- 使用者因此同時看到「路線折線 + 雨區回波」，達成「地圖疊雨區」需求。

### 快取
- 後端每 ~10 分鐘抓一次 CWA 雷達圖（CWA 更新頻率約每 10 分鐘），避免每次請求都打 CWA。
- 快取需可替換（記憶體 / 本機檔 / 未來換雲端物件儲存），呼應遷移原則。

## 6. 分階段開發

### Phase 1（MVP，本規格重點）— 即時降雨
完整可用的 App：
1. 輸入出發地、目的地（文字地址或地圖點選）
2. 規劃路線並在地圖上畫出
3. 疊上**當下**雷達回波
4. 執行雨區判斷，最上方顯示一句結論 + 標示有雨路段

**Phase 1 完成即為一個可自己騎車實用的 App。**

### Phase 2 — 沿途時間預測
- 依使用者預計經過各路段的時間（用路線的距離／預估車速推算到達各點的時刻），
  比對「那個時間點」的降雨短時預報（nowcast），而非只有當下。
- 資料源：CWA 短時／臨近降水預報相關產品（實作 Phase 2 時再確認具體資料集代碼）。
- 結論改為：「出發後約 X 分鐘、在 Y 路段會遇到雨」。

### 未來（非承諾）
- AI 建議 agent 層（LangGraph）：把雨區、溫度、風等整合成自然語言出行建議。
- 從 NUC 遷移到免費雲端（Plan A）。
- 機車專屬路線最佳化。

## 7. 技術棧（全開源／免費）

| 部分 | 選用 | 備註 |
|------|------|------|
| App 框架 | **Expo (React Native)** | 可直接跑自己手機、不必上架；用 Expo Go 或 dev build |
| 地圖顯示 | **MapLibre**（`@maplibre/maplibre-react-native`） | 開源地圖引擎 |
| 地圖圖磚 | **OpenFreeMap** 免費向量圖磚 | 無需金鑰；台灣 OSM 圖資完整 |
| 路線規劃 | **OpenRouteService** 免費 tier | 用 `driving-car` profile 代表機車；免費 key |
| 地址搜尋 | **Nominatim (OSM)** | 免費；注意其使用政策（加 User-Agent、限速） |
| 雨區資料 | **CWA 開放資料 `O-A0058`** | 免費註冊取得 key |
| 後端 | **Python FastAPI** | 使用者主力語言 |
| 後端部署 | Phase 1：NUC；未來：免費雲端 | 依環境變數切換 |

### 金鑰／設定清單（都放後端環境變數）
- `CWA_API_KEY` — CWA 開放資料授權碼
- `ORS_API_KEY` — OpenRouteService 金鑰
- `NOMINATIM_USER_AGENT` — Nominatim 要求的識別字串
- App 端：`BACKEND_BASE_URL` — 後端位置（換部署只改這個）

## 8. 後端 API 介面（草案）

> 皆為 App → 後端。實作時可依需要調整欄位命名。

### `POST /geocode`
- 輸入：`{ "query": "台北車站" }`
- 輸出：`{ "candidates": [ { "name": "...", "lat": 25.04, "lng": 121.51 }, ... ] }`

### `POST /route`
- 輸入：`{ "origin": {lat,lng}, "destination": {lat,lng} }`
- 輸出：`{ "polyline": [ [lat,lng], ... ], "distance_m": 12345, "duration_s": 1800 }`

### `POST /rain`
- 輸入：`{ "polyline": [ [lat,lng], ... ] }`
- 輸出：
  ```json
  {
    "verdict": "raincoat_recommended",        // 或 "no_raincoat_needed"
    "max_level": "moderate",                   // none/light/moderate/heavy
    "wet_segments": [ { "index": 12, "lat": ..., "lng": ..., "level": "moderate" } ],
    "radar_time": "2026-07-21T14:30:00+08:00",  // 雷達圖時間
    "overlay": { "image_url": "...", "bbox": [W,S,E,N] }  // 供地圖疊圖
  }
  ```

（`/route` + `/rain` 也可合併成一個 `/plan` 端點，減少往返；實作者自行決定。）

## 9. 成功標準（驗收）

Phase 1 視為完成，需滿足：
1. 能輸入起訖點，地圖上畫出一條路線。
2. 地圖上能看到半透明的即時雷達雨區疊圖，且與路線正確對齊。
3. 已知一條「明顯會經過雨區」的路線，App 正確給出「建議穿雨衣」並標出有雨路段；已知晴天路線給出「不需要穿雨衣」。
4. 所有外部金鑰只存在於後端環境變數，App 內無任何金鑰。
5. 後端可在 NUC 上跑起來，App（在使用者手機上）能連到它完成上述流程。

## 10. 給實作者的注意事項

1. **第一步先驗證資料源**：用 CWA Swagger 文件（https://opendata.cwa.gov.tw/dist/opendata-swagger.html）確認 `O-A0058` 各子項的實際回傳格式、影像網址、GeoInfo 欄位，再開始寫像素取樣。不要假設，先打一次 API 看實際回傳。
2. **地理範圍以回傳 metadata 為準**，第 5 節的經緯度是預期值／驗證基準，不要寫死。
3. **色階對照表要對照 CWA 官方**，並保留成可調設定，之後容易微調分級閾值。
4. **遷移原則是硬需求**：任何寫死的網址、路徑、金鑰都算 bug。
5. Nominatim / OpenRouteService 免費 tier 有速率限制，請加上合理的 User-Agent 與必要的節流。
6. 先把 Phase 1 做到「可自己騎車用」再談 Phase 2，不要提前實作時間預測。
